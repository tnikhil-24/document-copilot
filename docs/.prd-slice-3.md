## Problem Statement

Right now, every Document Copilot conversation ends in the same hardcoded reply: "Answer not yet implemented." Driftwood analysts can sign in, send messages, and see them persist (Slices 1-2), but the product does not yet do the one thing it exists for — answer questions about SEC filings with real, verifiable evidence.

The cost of getting this wrong is asymmetric. Driftwood's entire business is being right; a wrong but confident answer is worse than no answer at all. So the first slice that produces real LLM output must, from day one, make it structurally impossible for an unvalidated citation to reach an analyst's screen.

## Solution

Replace the stubbed reply with a real, grounded answer pipeline:

1. A PydanticAI agent receives the analyst's question and retrieves relevant filing passages itself, via a `search_filings` tool backed by pgvector semantic search (optionally scoped by company ticker and filing year) and a supplementary `read_chunk` tool.
2. The agent produces a structured answer: prose with inline `[1]`, `[2]` citation markers, a list of citations (marker + chunk ID), and an explicit flag for whether the corpus had sufficient evidence to answer at all.
3. Before anything is shown to the analyst, a grounding validator checks that every citation maps to a chunk the agent actually retrieved during this run, that markers and citations line up 1:1, and that the evidence flag is consistent with the citation list. The agent gets one chance to self-correct (via PydanticAI's retry mechanism) if validation fails.
4. Only once validation passes does the backend hydrate the real source passages (text + company/ticker/filing type/date/section) from the database — never from anything the model said about its own sources — stream the answer and citations to the browser, and persist both the message and its citations.
5. If the pipeline fails at any point (retrieval, generation, or validation never converges), the analyst sees a clean error and nothing is persisted — no half-written answers, ever.
6. In the browser, citation markers in the answer text are clickable and open a modal showing the exact source passage and its metadata, so an analyst can verify a claim in one click.

## User Stories

1. As an analyst, I want to ask a question about a filing and get a real, substantive answer instead of a stub, so that I can start using the tool for actual research.
2. As an analyst, I want every factual claim in the answer to carry a citation, so that I can verify it against the underlying filing.
3. As an analyst, I want to click a citation marker and see the exact passage it's based on, so that I can confirm the claim without searching the filing myself.
4. As an analyst, I want each citation to show the company, ticker, filing type, filing date, and section, so that I know precisely where in the original filing to look.
5. As an analyst, I want an honest "the corpus doesn't have enough evidence" response when I ask something outside the corpus, so that I'm never misled by a guess.
6. As an analyst, I want assurance that I will never see an answer containing a fabricated or unverified citation, so that I can trust every citation without double-checking the system itself.
7. As an analyst, if something goes wrong while generating a response, I want a clear error message rather than a partial or broken answer, so that I'm not confused about what happened.
8. As an analyst, I want citations to still render correctly — same markers, same source passages — when I reload a past conversation, so that my verification trail doesn't disappear on refresh.
9. As an analyst, I want questions that name a specific company and year (e.g., "Netflix's 2023 10-K") to reliably pull passages from that filing, so that single-document questions work well even before the system has more advanced retrieval.
10. As a backend developer, I want a retrieval module that performs pgvector semantic search with optional ticker/year filters, so that the agent can scope searches to the relevant filing(s) rather than the whole corpus.
11. As a backend developer, I want pgvector similarity search exposed as a Postgres RPC function callable through the existing Supabase client, so that retrieval uses the same DB-access pattern as the rest of the request path.
12. As a backend developer, I want a PydanticAI agent with explicit typed dependencies (`retriever`, `user_id`, `thread_id`) and a typed output (`GroundedAnswer`), so that the LLM boundary is testable in isolation from the rest of the system.
13. As a backend developer, I want the agent constrained so it can only cite chunks it actually retrieved during the current run, so that hallucinated citations are structurally impossible, not just discouraged by prompt wording.
14. As a backend developer, I want the grounding validator implemented as a pure, independently testable function, so that citation-validation logic can be unit tested without ever calling the LLM.
15. As a backend developer, I want the agent to get one self-correction attempt (via `ModelRetry`) when it produces an invalid citation set, so that an occasional model slip-up doesn't become a user-facing failure.
16. As a backend developer, I want a hard validation gate after the agent's run that returns a real 502 and persists nothing if the output is still invalid, so that "no half-message is persisted" is a structural guarantee, not a best effort.
17. As a backend developer, I want the full retrieve → agent → validate pipeline to complete before the SSE stream opens, so that a failure produces an actual HTTP 502 status rather than an in-band error after the client has already rendered untrusted text.
18. As a backend developer, I want source passages (text, company, ticker, filing type, date, section) hydrated from `document_chunks` after validation, so that the analyst-facing source data always reflects the database, never the model's own claims.
19. As a backend developer, I want validated citations persisted to `message_citations` with an explicit marker ordering, so that reloading a thread reconstructs the exact same `[1]`/`[2]` markers and source list.
20. As a frontend developer, I want a new streaming event/UI-message-part carrying validated source passages, so that the chat UI can render citations only once they're confirmed grounded.
21. As a frontend developer, I want inline `[1]`/`[2]` citation markers in assistant messages to be clickable, so that analysts can jump straight from a claim to its source.
22. As a frontend developer, I want a modal that shows a cited passage's full text and metadata, so that "click a citation, see the source" is satisfied end to end.
23. As a frontend developer, I want past messages loaded from `GET /threads/{id}` to include the same citation data as live responses, so that the citation UI doesn't differ between a fresh answer and a reloaded one.
24. As an operator, I want the agent's chat model and per-turn request limit defined in `app/config.py`, so that cost/latency/quality tradeoffs can be tuned without touching orchestration code.
25. As an operator, I want retrieval, agent, and validation failures to fail closed (502, nothing persisted), so that a degraded OpenAI or Supabase dependency cannot result in a misleading answer reaching an analyst.

## Implementation Decisions

**Turn flow — validate-then-replay.** `POST /chat/stream` no longer opens its `StreamingResponse` immediately. The route awaits the full pipeline first: persist the user message, run the agent, validate citations. If retrieval, the agent run, or validation fails (including after the agent's self-correction attempt), the route raises `HTTPException(502)` — no SSE stream is ever opened, and nothing beyond the user's own message is persisted. Only on success does the route open the `StreamingResponse`, replaying the validated `answer` text as token deltas (reusing the existing tokenization approach from the stub), then emitting the new citations event, then persisting the assistant message and its citations. This is the mechanism that makes "the user sees a 502 and no half-message is persisted" literally true.

**Conversation context — stateless per turn.** The agent operates only on the current user message; no prior thread history is passed as PydanticAI `message_history`. This matches the "naive retrieval" framing of this slice and avoids the complexity of reconstructing prior turns (including their citations) into agent message history. Multi-turn context is a candidate for a future slice.

**Retrieval module (`app/retrieval/`).** A `retriever` embeds the user's question with the existing OpenAI embedding model/dimensions, then calls a new Postgres RPC function `match_document_chunks(query_embedding, match_count, filter_ticker, filter_year)` through the user-scoped Supabase client (`client.rpc(...)`). The RPC performs cosine-distance ordering over `document_chunks.embedding` with optional `WHERE` filters on `chunk_metadata->>'ticker'` and `chunk_metadata->>'year'` (NULL filter = no constraint), returning full chunk content plus metadata. Default `top_k = 8`. This keeps request-path DB access on a single pattern (Supabase client), consistent with `app/database/chats.py`, and the same RPC-function approach extends naturally to Slice 4's hybrid search.

**Agent module (`app/assistant/`).**
- `agent.py`: a PydanticAI agent using `gpt-4o-mini`, with a per-turn `request_limit=5` (covers `search_filings`, an optional `read_chunk`, and the final structured response). Exceeding the limit without producing a valid `GroundedAnswer` is treated as an agent failure (502).
- `deps.py`: a dependency object bundling the Supabase client, `user_id`, `thread_id`, the retriever, and a mutable registry of chunk IDs retrieved so far during this run.
- `outputs.py`: 
  - `Citation { marker: int, chunk_id: UUID }` — minimal, model-produced.
  - `GroundedAnswer { answer: str, citations: list[Citation], has_sufficient_evidence: bool }` — `answer` contains inline `[1]`, `[2]`-style markers matching `citations[].marker`.
  - `SourcePassage` — backend-hydrated only (chunk_id, marker, ticker, company_name, filing_type, filing_date, section, full chunk content). Never produced by the model.
- `instructions.md`: encodes the grounding contract — cite every factual claim using retrieved chunks only, set `has_sufficient_evidence=False` and leave `citations` empty when the corpus doesn't support an answer, never give investment advice or recommendations, keep answers concise.
- `search_filings(query, ticker=None, year=None, top_k=8)` tool: calls the retriever, returns full chunk content + metadata, and records each returned `chunk_id` in the deps registry. Instructions tell the agent to extract company/year hints from the question when present.
- `read_chunk(chunk_id)` tool: re-fetches a chunk's full content; only succeeds for chunk IDs already present in the deps registry (a chunk must have come from `search_filings` first). For this slice's single-fact questions it's expected to be used rarely, if ever.

**Grounding validator (`app/grounding/validator.py`).** A pure function `validate_citations(answer: GroundedAnswer, retrieved_chunk_ids: set[UUID]) -> None`, raising on failure, checking:
- Every marker referenced in `answer.answer` has a corresponding `Citation`, and vice versa (no orphaned markers or unused citations).
- Every `Citation.chunk_id` is in `retrieved_chunk_ids`.
- `has_sufficient_evidence` is consistent with `citations`: `True` requires a non-empty, fully-valid citation list; `False` requires an empty citation list.

This function is wired into the agent via `@agent.output_validator`. On failure, it raises `pydantic_ai.ModelRetry` with a message describing which markers/chunk IDs are actually valid, giving the model one in-loop chance to correct itself (consuming part of the `request_limit` budget). After the agent run completes, the orchestrator calls the same `validate_citations` once more as the final hard gate — if it still fails, the turn fails with 502 and nothing is persisted.

**Source passage hydration & persistence.** After validation passes, the orchestrator builds `SourcePassage` objects for each citation directly from the retrieved `document_chunks` rows already held by the deps registry (full content + `chunk_metadata` fields: ticker, company_name, filing_type, filing_date, section). These are what gets streamed to the client and persisted — the model's own output never reaches the analyst as "source" data.

**Schema changes (Alembic migration).**
- New SQL function `match_document_chunks` (cosine-distance kNN with optional ticker/year filters), added via `op.execute(...)`.
- `MessageCitation` gains a `marker: Mapped[int]` column, so that reloading a thread can reconstruct the same `[1]`/`[2]` ↔ source-passage mapping that was shown live. `message_citations.excerpt` continues to store the full chunk content (consistent with `search_filings` returning full content).

**Config (`app/config.py`).** New setting `openai_chat_model: str = "gpt-4o-mini"`.

**Streaming contract.** `ChatStreamEvent` gains a `CitationsEvent { passages: list[SourcePassage] }`, encoded as a single custom AI SDK UI message part: `{"type": "data-citations", "data": {"passages": [...]}}`, emitted once after the `text-end` event and before `finish-step`/`finish`. `to_ui_message` (used by `GET /threads/{thread_id}`) is extended to join `message_citations` with `document_chunks` for assistant messages and append the same `data-citations` part, so reloaded history renders identically to a live response.

**Frontend.** `ChatMessage` parses `[1]`/`[2]` markers in the message's text part and renders them as clickable spans; clicking opens a shadcn `Dialog` showing the corresponding `SourcePassage`'s content, ticker, company name, filing type/date, and section. This is intentionally basic — Slice 5 polishes the same modal (hover preview, EDGAR link, layout).

## Testing Decisions

Tests verify externally observable behavior of each module's public interface, consistent with the pattern established in Slice 1.

- **`app/grounding/validator.py`**: unit tests for `validate_citations` covering — valid grounded answer, valid refusal (empty citations + `has_sufficient_evidence=False`), citation referencing a chunk ID outside the retrieved set, marker present in `answer` text with no matching `Citation` (and vice versa), and `has_sufficient_evidence=True` with empty citations. This is the highest-value coverage in the slice — it's the actual trust boundary.
- **`app/retrieval/retriever.py`**: unit tests for query construction (embedding call, RPC parameters including ticker/year filter handling and `top_k`), with the Supabase client mocked at the boundary.
- **`app/assistant/agent.py`**: unit tests using PydanticAI's `TestModel`/`FunctionModel` to simulate both well-formed `GroundedAnswer` outputs and adversarial ones (citing an unretrieved chunk ID, mismatched markers, inconsistent `has_sufficient_evidence`) — verifying the `output_validator` raises `ModelRetry` and that the agent's tool constraints (e.g., `read_chunk` rejecting a chunk ID never returned by `search_filings`) hold. Per `backend/CLAUDE.md`, mocking the LLM here is paired with exercising the grounding contract itself, not just orchestration plumbing.
- **Orchestrator (`run_turn` / `stream_chat`)**: unit tests for both the success path (events emitted in order: text deltas, then `CitationsEvent`, then completion; user and assistant messages plus citations persisted) and the failure path (any pipeline failure → `HTTPException(502)`, only the user message persisted, no assistant message or citations written).
- **Integration tests** (`@pytest.mark.integration`, require live OpenAI + Supabase): the full retrieve → agent → validate → persist turn run against the brief's single-document questions (#3, #4, #5, #10), confirming cited, grounded answers; plus a known out-of-corpus question confirming an honest refusal.
- **Frontend**: no automated tests, per `frontend/CLAUDE.md`. Manually verify in-browser: a single-fact question renders a cited answer with clickable markers; clicking a marker opens the modal with correct passage/metadata; an out-of-corpus question renders a refusal with no citations; reloading the thread renders identical citations.

## Out of Scope

- Hybrid search (full-text + RRF fusion) and the `read_surrounding_chunks` tool — Slice 4.
- Multi-document / comparative / cross-year questions (brief questions #1, #2, #6, #7, #8, #9) — Slice 4.
- Multi-turn conversational context for retrieval (passing thread history to the agent) — deferred indefinitely, not part of this slice.
- Citation UI polish: hover preview, polished modal styling, "Verify in EDGAR" links — Slice 5.
- Loading-state copy ("retrieving documents...", "generating answer...") and other empty/loading/error UX polish — Slice 6.
- Structured logging pass across slices — Slice 6.
- Deep semantic/entailment grounding (verifying the model's *claim* is actually supported by the *passage text*, beyond referential chunk-ID validation) — not attempted in this slice; "grounded" here means referentially grounded.

## Further Notes

This PRD covers Slice 3 of the 7-slice vertical-slice restructuring described in `docs/todos.md`. It is the first slice where a real LLM produces a real answer, and its central architectural commitment — validating fully before any byte streams to the client — is what makes "no demoable state where an unvalidated citation can appear" a structural property of the system rather than a hopeful prompt instruction.

The definition of done is a literal walkthrough against the live Railway deployment: ask one of the brief's simpler single-fact questions (e.g., "What did Netflix's 2023 10-K say about content costs?") and get a real, validated, cited answer; click a citation and see the actual source passage with company/ticker/filing date/page; ask something outside the corpus and get an honest refusal, never a guess; and confirm that if the LLM or grounding step fails, the analyst sees a 502 and no half-message is persisted.

The `match_document_chunks` RPC function and the agent/deps/output module shapes established here are deliberately reusable: Slice 4 adds a sibling RPC for lexical search and fuses results in Python (RRF), without changing the agent's tool-calling contract or the grounding validator's interface.

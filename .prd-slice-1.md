## Problem Statement

Driftwood analysts need to use Document Copilot through a browser, signed in with their Driftwood email. Right now there's no working chat loop at all: no auth verification, no message persistence, no streaming endpoint, and no frontend. Before any retrieval or LLM work can be meaningfully built or demoed, the team needs proof that a logged-in analyst can send a message and get a response that survives a page reload — end to end, with no mocked layers standing in for real systems.

## Solution

Build the first vertical slice of Document Copilot: an analyst signs in with their Driftwood email via Supabase Auth, lands in an auto-created chat thread, sends a message, watches a (currently stubbed) assistant reply stream back token-by-token, and finds the conversation intact after a refresh. Every hop — auth, persistence, streaming, rendering — is real; only the assistant's actual answer is a placeholder, to be replaced in a later slice. The slice also stands up the Railway deployment for both services, so the working loop is verifiable in production, not just local.

## User Stories

1. As an analyst, I want to sign in with my Driftwood email, so that I can access Document Copilot with my own identity.
2. As an analyst, I want my session to persist across page reloads, so that I do not have to log in every time I open the app.
3. As an analyst, I want to log out, so that I can end my session on a shared machine.
4. As an analyst with an expired or invalid session, I want to be redirected to the login page, so that I understand I need to sign in again.
5. As an analyst, I want to land in a conversation when I open the app, so that I can immediately start asking questions.
6. As an analyst, I want to type a message and send it, so that I can ask the assistant a question.
7. As an analyst, I want to see my message appear immediately after sending it, so that I have confirmation it was received.
8. As an analyst, I want to see the assistant's reply stream in token-by-token, so that I get a sense of progress rather than staring at a blank screen.
9. As an analyst, I want to see a loading indicator while waiting for a response, so that I know the system is working.
10. As an analyst, I want my conversation to still be there when I refresh the page, so that I do not lose my work.
11. As an analyst, I want a friendly error message if something goes wrong, so that I am not confused by a blank screen or a raw error.
12. As a backend developer, I want JWT verification at the FastAPI boundary, so that every protected route can trust the current-user dependency without re-implementing auth checks.
13. As a backend developer, I want a profile record created automatically on first login, so that the rest of the system can reference a stable internal user ID.
14. As a backend developer, I want a persistence layer for threads and messages, so that the orchestrator and routes do not need to know about the underlying database details.
15. As a backend developer, I want an orchestrator module that coordinates one chat turn, so that a later slice can swap the stub for a real LLM agent without changing the route layer.
16. As a backend developer, I want typed Pydantic models for streaming events, so that the frontend and backend agree on a stable wire format.
17. As an operator, I want both frontend and backend deployed to Railway with secrets injected via the Railway UI, so that the slice is verifiable in production, not just locally.
18. As an operator, I want migrations run via the session connection rather than the pooler, so that schema changes apply reliably.
19. As an analyst who tries to access a thread that does not exist, I want a clear not-found response, so that the app does not hang or show a confusing error.
20. As an analyst who sends a request with an invalid or missing token, I want a clear unauthorized response, so that I understand I need to sign in.

## Implementation Decisions

- Auth boundary: JWT verification happens once, at the FastAPI dependency layer (a current-user style dependency), not duplicated per route. On first successful verification, the backend creates a profile row linked to the Supabase auth user ID if one does not exist yet — this keeps the rest of the system referencing a stable internal user ID rather than the raw Supabase auth ID.
- Single auto-created thread: this slice deliberately does not build thread management (list, create, switch — that is Slice 2). Instead, opening the app auto-creates or resumes a single thread for the user, keeping the slice's surface area minimal while still exercising real persistence.
- Persistence layer for chats: exposes a small CRUD-shaped interface — create thread, append message, fetch message history — so the orchestrator and routes depend on a stable contract rather than raw database calls. This is the seam Slice 2 extends (thread listing) and Slice 3 writes through (citations).
- Orchestrator seam: a run-turn style interface (given a thread and messages, emit a stream of events) coordinates one chat turn against a hardcoded stub response in this slice. The interface is designed so Slice 3 can replace the stub with a real PydanticAI agent invocation without changing the route layer or the streaming contract.
- Message conversion: a dedicated module converts between the Vercel AI SDK's wire message format and the backend's internal message representation, isolating that translation from orchestration logic.
- Streaming contract: Pydantic models define the streaming event types (text delta, completion, error) up front, so the frontend useChat integration has a stable contract to build against — even though the content behind it is a stub for now.
- Frontend layering: env validation module (reads and validates required Vite env vars), Supabase browser client module, an HTTP wrapper that injects the bearer token and converts failures to typed errors, and a product-level API module for loading/creating threads and sending messages. Each layer is a small, swappable seam — the API module is the only thing chat components talk to.
- Chat UI: built on the Vercel AI SDK useChat hook, transport pointed at the chat-stream endpoint, with the Supabase access token injected via the transport's headers callback (per the shape already documented in docs/architecture.md).
- Error handling baked in, not deferred: 401 for invalid, missing, or expired tokens; 404 for a missing thread. These are implemented as part of this slice — they are the natural error paths this slice's auth and persistence code expose, not a later hardening pass.
- Deployment: both Railway services (frontend static build, backend FastAPI/Uvicorn) are stood up as part of this slice, with all secrets injected via the Railway secrets UI and migrations run through the session connection string rather than the transaction pooler — this derisks infra issues while the app is still simple.

## Testing Decisions

Tests should verify externally observable behavior — the inputs and outputs of a module's public interface — not internal implementation details like specific SQL queries or HTTP call counts.

- Chat persistence layer: integration tests against a real test database verifying create-thread, append-message, and fetch-history behave correctly, including ordering and isolation between threads. A regression here would silently corrupt chat history, so this is high-value to cover early.
- Orchestrator and message conversion: unit tests for the turn-coordination logic (given a thread and messages, does it emit the expected event sequence against the stub) and for AI-SDK-to-internal message format conversion in both directions. This is the seam Slice 3 will swap from stub to real agent — tests here protect that swap from breaking the route contract.
- Auth verification: unit and integration tests for token validation (valid, expired, malformed, missing) and for first-login profile creation (creates exactly once, does not duplicate on subsequent logins). This is a security-relevant boundary where silent failures are easy to miss.
- Frontend HTTP and API layers: unit tests for auth-token injection, typed error conversion on failure responses, and the shape of product-level API calls — the seam between the React app and the backend.
- Out of scope for automated tests in this slice: raw streaming wire behavior and UI rendering — these are better validated through the manual demo script (the slice's definition of done) than through brittle stream-mocking tests.

Prior art: there is no existing test suite in the backend or frontend yet — this slice establishes the testing pattern later slices will follow.

## Out of Scope

- Thread list, thread creation UI, switching between conversations (Slice 2)
- Any real LLM-generated answer — the assistant response is a hardcoded stub in this slice (real grounded answers land in Slice 3)
- Retrieval, citations, grounding validation (Slices 3-4)
- Citation UI, EDGAR links (Slices 3, 5)
- Empty states, rich loading copy such as "retrieving documents...", responsive polish (Slice 6)
- Corpus ingestion (Track A — runs in parallel but is a separate, non-user-facing prerequisite)
- 403 responses for cross-user thread access — not reachable until Slice 2 introduces multiple threads per user
- User docs, FAQ, feedback channel, metrics baseline (Slice 7)

## Further Notes

This PRD covers Slice 1 of a 7-slice vertical-slice restructuring of docs/todos.md. The original plan organized Phases 2-13 as horizontal layers (build all of auth, then all of the API, then all of the frontend, then all of retrieval, then all of grounding, and so on), which deferred anything demoable until very late. Slice 1's entire purpose is to prove the wiring works end to end with zero mocked layers, so every later slice (grounded answers, retrieval quality, citations) can be built and demoed against a real, working chat loop rather than imagined plumbing.

The definition of done is a literal walkthrough performed against the live Railway deployment, not just localhost: log in, land in an auto-created thread, send a message, watch a stubbed reply stream in token-by-token, refresh the page, and confirm the conversation persists. An invalid token returns 401; a request for a missing thread returns 404.
